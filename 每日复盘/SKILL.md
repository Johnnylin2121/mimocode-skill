# 每日复盘工作流

## 核心原则

1. **预期-验证闭环**：每日复盘提出假设，次日早盘验证，形成闭环
2. **数据驱动**：所有判断必须有数据支撑，不用感觉代替事实
3. **连贯性验证**：每日复盘必须回顾前日预测，判断正确/偏差/遗漏
4. **可验证假设**：每次复盘结束必须提出至少一个可验证假设
5. **复盘与交易计划分离**：复盘只分析市场，不关联持仓；交易计划只定义执行规则
6. **三日窗口**：复盘基于今天 + 前两个交易日，不多不少
7. **计划你的交易，交易你的计划**：入场前所有决策已定好，盘中只执行不决策

## 工作流三阶段

### 阶段一：盘前准备（9:00前）

**输入**：
- 前日复盘文件（`D:\OneDrive\ObsidianVault\30-Resources\每日复盘\`）
- 当日财经早读（`D:\OneDrive\ObsidianVault\30-Resources\财经早读\`）
- 前日盘前预测和交易观察清单（`D:\OneDrive\ObsidianVault\30-Resources\盘前预测\`）
- 交易计划文件（`D:\OneDrive\ObsidianVault\30-Resources\交易计划\`）

**步骤**：
1. 读取前日复盘的"明日关注点"和"可验证假设"
2. 读取当日财经早读，提取关键信息
3. **审查交易计划**：读取所有状态为"等待入场"或"持仓中"的交易计划，检查：
   - 入场条件是否接近触发（接近触发时主动提醒用户）
   - 已持仓计划的前提是否还成立（市场方向是否发生重大变化）
   - 失效条件是否已被触发
4. 制定观察清单和盘前预测，写入本地文件（`30-Resources/盘前预测/`）

**交易计划审查输出**：
```markdown
### 交易计划审查
- [计划名称]：[状态] — [审查结论]
  - 入场条件：[是否接近触发]
  - 前提是否成立：[是/否，理由]
  - 操作建议：[继续等待 / 有介入机会 / 建议失效]
```

**观察清单结构**（本地文件）：
```
第一层：市场整体量能
  - 成交额判断标准（>2.8万亿=确立；2.5-2.8万亿=试探；<2.5万亿=萎缩）

第二层：板块龙头量能验证
  - 各方向龙头个股+判断标准

第三层：科技分化信号
  - 关键标的+判断标准

决策树
  - 放量>2.8万亿？是→连板=确认；否→缩量→观望
```

### 阶段二：盘中验证（10:00 / 13:00）

**时机**：
- 早盘验证：10:00-10:30
- 午盘验证：13:00-13:30（如有反转）

**步骤**：
1. 拉取实时数据（指数+观察标的+板块资金流向）
2. 逐条对照观察清单验证
3. **交易计划跟踪**：检查是否有交易计划的入场条件接近触发，如有则提醒用户
4. 更新本地交易观察清单（添加"盘中验证 HH:MM"章节）

**验证结果格式**：
```markdown
## 盘中验证 HH:MM

- 两市成交额~XXX亿，推算收盘X.X万亿，[试探/确立/萎缩]
- [方向1]：[结论]（龙头个股数据）
- [方向2]：[结论]（龙头个股数据）

### 交易计划跟踪
- [计划名称]：入场条件 [接近/未接近] 触发

### 操作决策
- [决策]：[原因]
```

**提醒规则**：
- 入场条件满足80%以上 → 主动提醒"有介入机会，是否制定/执行交易计划？"
- 已持仓计划触发止损线 → 立即提醒
- 市场出现重大方向变化可能影响已有计划前提 → 提醒审查

---

### 阶段三：盘后复盘（15:00后）

**输入**：
- 当日完整交易数据
- 盘中验证结果
- 前日复盘文件
- 交易计划文件

**输出**：
- 更新前日复盘的"今日验证"章节
- 创建当日复盘文件
- **审查所有交易计划**：复盘完成后，检查每个交易计划的前提是否因当日行情而改变

**关键规则**：
> 早盘验证写入前日复盘，形成"预期-验证"闭环。当日复盘在收盘后创建，因为早盘仍有反转不确定性。

**复盘后交易计划审查**：
1. 读取 `30-Resources/交易计划/` 下所有状态非"已完成"或"已失效"的计划
2. 对照当日复盘结论，检查计划的"决策依据"是否仍成立
3. 检查"失效条件"是否已被触发
4. 更新计划的审查记录：
   - 前提成立 → 记录"审查通过，继续等待"
   - 前提动摇 → 记录"审查警告：[具体变化]，建议关注"
   - 前提不成立 → 标记计划失效，移入归档
5. 如有新建交易计划的需求（观察清单中出现新机会），提醒用户

## 常见错误

1. **时间混淆**：Notion API返回UTC时间，需+8小时转换为东八区
2. **早盘验证写错位置**：应写入前日复盘的"今日验证"，不是当日复盘
3. **图片占位符**：markitdown转换会生成`![](data:image/jpeg;base64...)`，需清理
4. **JSON编码**：PowerShell的ConvertTo-Json对Notion API会出错，需用UTF-8文件方式
5. **复盘中关联交易计划**：复盘只分析市场，不关联持仓。交易计划的审查在复盘完成后独立进行
6. **盘中临时修改交易计划**：交易计划一旦制定，盘中只执行不修改。修改只能在盘后审查时进行
7. **交易计划制定无人工参与**：MiMo可以提醒有介入机会，但交易计划必须经用户确认后才创建
